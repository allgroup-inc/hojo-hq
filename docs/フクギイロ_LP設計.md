# フクギイロ LPサイト設計書 — エンジニアリング全項目

作成日: 2026-07-22 / 実装: site/fukugiiro/(このリポジトリ内・既存Pagesデプロイに同乗)
目的: 24/365円環(全体像①〜②)の入口。**LPの唯一の仕事は「診断を始めさせること」**。

---

## 1. 情報設計(IA)— 1ページ・1CTA

| セクション | 内容 | 狙い |
|---|---|---|
| ヒーロー | 「あなたの世帯、いくらもらい忘れてる?」+診断CTA | 3秒で自分ごと化 |
| 実受給カウンター | フクギイロ経由の受給確認額(自動更新) | 社会的証明。0円の間は「掲載制度数・カバー市町村数」を表示 |
| 使い方3ステップ | 診断3分→書類チェックリスト→入金確認まで伴走 | 「情報サイトではなく伴走」の差別化 |
| もらい忘れ例 | 代表制度カード(児童手当・住居確保給付金等、DB連動) | 具体性。全て「可能性」表記+一次ソースリンク |
| 信頼セクション | 正確性ポリシー(一次ソース照合・要確認表示)・運営会社・出典(PDL1.0) | 行政情報を扱う者の顔を見せる |
| FAQ | 「無料?」「なぜ無料?」「代行してくれる?」(=しない、を明記) | 不安と法的誤解の先回り |
| フッター | 個人情報保護方針・運営会社・姉妹サイト(企業のミカタ) | コンプライアンス |

CTAは全セクションで**診断1つだけ**(CV単一原則のLP版)。LINE登録は診断結果画面の出口に置き、LPには置かない(診断を挟む方が登録の質が上がる=ツナグ設計思想)。

## 2. 技術選定(エンジニア視点の決定と理由)

| 項目 | 決定 | 理由 |
|---|---|---|
| スタック | 静的HTML+素のCSS/JS(フレームワークなし・依存ゼロ) | hojo-hq本体と同一。ビルド工程ゼロ=壊れる場所がゼロ。エージェントが安全に編集できる |
| ホスティング | GitHub Pages(既存update.ymlのsite/デプロイに同乗) | 追加費用ゼロ・実証済みパイプライン |
| パス | site/fukugiiro/index.html(将来: fukugiiro.jp独自ドメイン+CNAME) | ブランド分離しつつ資産共有。独自ドメインは小柳さん決裁事項 |
| データ結合 | data/fukugiiro/seido.json をビルドなしでfetchし制度カード・件数を描画 | DB更新=サイト更新。二重管理なし |
| ブランド | CSSカスタムプロパティでトークン化(--fg-primary等)。現在は仮色 | ロゴ・配色データ受領後に**トークン1箇所の差し替えで全体反映** |
| フォント | システムフォント+Noto Sans JP(font-display: swap) | CLS/LCP対策。Webフォントはサブセット化まではsystem優先 |
| 診断 | 完全クライアントサイド(JS)。ロジックJSONはDB由来・eval合格版のみ配置 | サーバー不要=個人情報がサーバーに渡らない(保護方針と整合) |
| 受給報告 | LINE開設までは受付導線を出さない(虚偽動線を作らない) | 正直さ優先。開設後にLINE経由で実装 |

## 3. 性能予算(Performance Budget)— CIで守る数値

- 初期表示合計 **150KB以下**(画像除く)/ HTML+CSS+JS それぞれ50KB以下
- LCP 2.5秒以内(3G相当)/ CLS 0.1未満 / 画像はWebP+width/height明示+lazy
- 外部リクエスト: seido.json 1本のみ(アナリティクス導入時も+1本まで)
- 検査: fukugiiro-ci.ymlにサイズ検査ステップを追加(超過でCI落ち=方程式4の適用)

## 4. SEO・構造化データ

- title/meta description/OGP(シェア画像はデザインパイプラインで生成)
- JSON-LD: FAQPage(FAQセクション連動)+Organization
- sitemap.xml・robots.txt(既存site/と統合)
- LPは指名検索+SNS流入の受け。**ロングテールSEOは別途programmatic pages(M4〜)が担当**。LPに全部載せない

## 5. 計測(プライバシーファースト)

- 第1段階(現在): UTMパラメータ+診断開始/完了イベントのみ。**Cookieなし・外部送信なし**(イベントはlocalStorageで診断完了率のみ自己計測→診断結果画面で任意送信は将来)
- 第2段階(要守り部審査+小柳さん決裁): Cookieレス計測ツール導入(GA4のCookieレス設定 or Plausible系)。導入までは「SNS側のリンククリック数×診断完了率」で代替
- 計測項目の定義: 訪問→診断開始率 / 診断完了率 / (LINE開設後)登録率30%目標

## 6. アクセシビリティ(高齢者・スマホ弱者への配慮=ユーエックスさん要件)

- 基本文字サイズ18px・行間1.8・コントラスト比4.5:1以上
- タップ領域44px以上 / フォーム要素にlabel / セマンティックHTML(main/nav/section+h1階層)
- ふりがな対応は第2段階(自治体サイト水準を目標)。「やさしい日本語」を本文の標準とする
- キーボード操作・スクリーンリーダーで診断が完了できること(eval項目に追加)

## 7. 運用・監視

- デプロイ: mainマージ→update.ymlが自動デプロイ(ステージング=ブランチのPRプレビューはHTMLアーティファクトで確認)
- 監視: 既存healthcheck.pyにfukugiiro/index.htmlとseido.json鮮度の検査を追加(障害時は自動Issue)
- 404/エラー: seido.json取得失敗時は静的フォールバック文言(カウンター非表示・診断は動く)
- 変更フロー: LP文言変更もPR+CI+金曜承認(L2)。禁止表現evalはLP文言にも適用

## 8. 三名体制議論(LP設計)

- スイシン: 1ページ1CTA・依存ゼロ・DB直結は最速で正しい。即実装可
- ウタガイ: カウンター0円のまま公開すると「実績なし」が逆効果。→ 採用: 受給確認額が10万円を超えるまでカウンターは「掲載制度数・カバー領域数」を主表示にする切替ロジックを実装
- ベッカイ: LPを作らず「診断そのものをトップページにする」別解もある(Turn2usはこれに近い)。→ 一部採用: ヒーローCTAは診断へ直行、LPは診断の「包み紙」に徹する。診断完了率が高ければ将来診断ファーストに転換
- 裁定: 上記2点を反映して実装開始。ロゴ・配色はトークン差し替え前提で仮色進行
