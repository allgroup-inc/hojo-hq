# フクギイロ 自動化経営の方程式 — 世界のClaude Code自動化ビジネスからの実践的抽出

作成日: 2026-07-22 / 親文書: フクギイロ構想_可処分所得1万円計画.md、フクギイロ_月次実行計画.md
目的: 世界中のClaude Code/AIエージェント自動化ビジネスの成功パターンを「方程式」として抽出し、フクギイロの設計・月次計画に1対1で埋め込む。

---

## 第1部 達成の方程式(7本)

世界の実例(Pieter Levels 年商$3.1M・利益率90%超を1人で運営 / Marc Lou 17連敗後にbuild in publicで月$45K超 / Base44 コード9割Claude製で6ヶ月→$80M売却 / 企業のClaude Code業務自動化で業務時間30〜70%削減の再現例多数)から抽出した共通構造。

### 方程式1: 利益 = 単価 × 件数 × 継続率 ÷ 人間時間
**「人を増やして件数を増やす」を「自動化して人間時間を減らす」に置き換えた者だけが利益率90%に到達する。**
- Levelsの核心は「複数の百万ドル事業をcronが回している」こと。人間は例外処理だけ
- フクギイロ適用: 全収益商品(送客・サブスクレポート・OEM)は「Claude生成+人間承認」で限界費用ほぼゼロの構造のものしか売らない。人間の工数が線形に増える商品(個別コンサル等)は設計段階で禁止

### 方程式2: 流通 > 製品(Distribution first)
**成功者は例外なく「作ってから売る」ではなく「売れる導線の中に作る」。**
3つの実証済み戦術をそのまま輸入する:
1. **Engineering as Marketing(無料ミニツールが最強の広告)**: Marc Louの定石。→ フクギイロの「世帯診断=いくらもらい忘れてる?」はまさにこれ。加えて「もらい忘れ計算機」「保育料いくら変わる?計算機」等の埋め込み可能ウィジェットを量産する(OEM商品の種にもなる)
2. **Programmatic SEO(掛け算でページを自動生成)**: 「制度 × 41市町村 × ライフイベント」= 数千ページをClaude Codeで自動生成・自動更新。「宜野湾市 出産 給付金」のようなロングテール検索を面で取る。検索流入目標(KGI内訳200社/世帯)の主武器
3. **Build in Public(数字の透明公開が信頼と拡散を生む)**: 実受給額カウンター・週次実績のSNS公開はbuild in publicの沖縄版。「夫婦2人+AIで県民運動」というストーリー自体がメディア・講演収益(柱5)になる

### 方程式3: 自動化はダイヤル、オン/オフではない
**2026年の運用標準: タスクごとに自律度を段階設定し、リスクが集中する所にだけ人間の承認を置く。**

| 自律度 | 対象 | 例 |
|---|---|---|
| L4 全自動 | 可逆・低リスク・大量 | 収集クロール、下書き生成、集計、社内レポート |
| L3 自動実行+事後監査 | 可逆・中リスク | 検証済み制度の定型更新、締切アラート配信 |
| L2 人間承認後に実行 | 不可逆・対外 | 新規制度の公開、SNS投稿、法人向けレポート送付、プレスリリース |
| L1 人間のみ | 金・法務・契約・謝罪 | 請求、契約締結、個人情報の取扱変更、事故対応、決裁上申 |

- 全件承認は「摩擦を生むだけで安全にならない」(Anthropic)。毎週金曜の一括承認バッチ(週次リズム)はこの思想の実装
- フクギイロ適用: 上表をそのまま運用規程にする。L2→L3への昇格は「eval合格率+事故ゼロ3ヶ月」を条件に守り部が審査

### 方程式4: evalなき自動化は必ず事故る
**成功している運用は全て「観測→失敗をテスト化→ポリシー更新」のループ(Agent Development Lifecycle)を回している。**
- フクギイロ適用(最重要領域=正確性):
  - 収集・構造化の**回帰テスト**: 過去に正解が確定した制度データをゴールデンセットにし、パイプライン変更のたび自動検証
  - 診断ロジックの**トラジェクトリeval**: 「この世帯属性ならこの制度が出るべき」のテストケースをClaudeが制度DBから自動生成→ケンショウ(検証部)が承認→CI で毎日実行
  - **事故のテスト化**: 誤情報・誤配信が起きたら、修正より先に「同じ事故を検出するテスト」を追加する。これを守り部の事故対応手順に明記
  - 生成系と検証系は**必ず別エージェント**(自己採点をさせない)

### 方程式5: 1事業・複数プロダクト・単一資産
**LevelsもLouも「ポートフォリオ」だが、成功物件は全て同じ資産(観客・データ・テンプレ)の再利用。ゼロから毎回作っていない。**
- フクギイロの単一資産 = **「沖縄で一番正確な制度DB+受給実績データ」**
- そこから派生するプロダクト群: ①県民向け診断(無料・集客) ②締切アラート(無料・継続接点) ③法人サブスクレポート(課金) ④OEM診断ウィジェット(課金) ⑤匿名統計・政策提言パッケージ(信用・行政接続) ⑥programmatic SEOページ群(流入)
- **新プロダクトの判定基準: 「制度DBの再利用か?」がNoなら作らない**(1年目)

### 方程式6: 無人経営基盤 = ヘッドレス + cron + リモート制御
**技術スタックの標準形が確立している。**
- `claude -p`(ヘッドレス)+ GitHub Actions cron: 収集・検証・生成・レポートの無人実行(hojo-hqで実証済み。1日2回→フクギイロでは用途別に増設)
- サブエージェント並列化: 41市町村の収集を市町村ごとに並列サブエージェントで実行し、検証エージェントが合流させる(split-and-merge)
- フック(hooks)による守り部ゲート: 「公開ディレクトリへの書き込み=人間承認必須」等をコードで強制(規律を意志力でなく仕組みにする)
- リモート制御/定期タスク: 夫婦のスマホから承認・指示できる体制(面談中でも経営が止まらない)
- コスト感覚: この種の無人パイプラインの運用コストは月額数千〜数万円で、パート1人分(数十万円)を置換する — 実例でも$15-40/月のトークン費で$1,500-2,500/月の分析業務を代替

### 方程式7: 週次の複利 = 出荷频度 × 学習速度
**Louの「17連敗→成功」が示す通り、勝ち筋は当てることではなく試行回数と学習の速さ。**
- 毎週必ず何かを出荷する(新制度・新ページ・新投稿・改善)。Claude Codeがあるため出荷の限界費用は人間の「承認30分」だけ
- 月次レビューで「施策を1つだけ変える」ルール(月次実行計画・指針9)はこの方程式の統制側。無秩序な試行はしない
- 失敗の記録を資産化: 撤退した施策・落ちた商談の理由をリポジトリに残し、エージェントの判断材料(CLAUDE.md/名簿)に還流する

## 第2部 アンチパターン(世界の失敗から)

1. **自動化の自動化に溺れる**: パイプラインいじりが楽しくて顧客に会わない。→ 火〜木は面談日と固定(週次リズム)で強制
2. **evalなしで公開系を自動化**: 誤情報事故→信頼全損。→ 方程式4のCIを先に作る(M1〜M2で構築、診断ローンチのM3より前)
3. **全部を人間承認にする**: 承認疲れで確認が形骸化し、かえって事故る。→ 自律度ダイヤル(方程式3)
4. **1年目からプロダクト多角化**: 資産再利用でないものに手を出して全部中途半端。→ 方程式5の判定基準
5. **広告先行**: 信頼資産(既存顧客・紹介)より先に広告を打つと、CACが利益構造を壊す。→ 指針7(広告費ゼロ)
6. **ストーリーなき透明化**: 数字だけ公開しても拡散しない。「県民の家計に防風林を」の物語と実名の声(指針8)をセットで

## 第3部 フクギイロ実装アーキテクチャ(Claude Code具体設計)

```
[情報源] 国API(jGrants)/県・41市町村サイト/広報紙PDF
    │  ① 収集cron(06:00/18:00 JST、市町村別サブエージェント並列)
    ▼
[構造化] Claude API → 個人向けスキーマJSON(ライフイベント・世帯要件・概算額)
    │  ② 検証エージェント(生成系とは別)が一次ソース突合 + ゴールデンセット回帰テスト(CI)
    ▼
[単一資産] 制度DB + 受給実績データ(リポジトリ管理・全変更履歴が監査ログ)
    │
    ├─③ 静的サイト生成(GitHub Pages) + programmatic SEOページ群(制度×市町村×ライフイベント)
    ├─④ LINE診断ロジック自動生成 → トラジェクトリeval合格後にデプロイ(L2承認)
    ├─⑤ 締切アラート配信(L3: 自動+事後監査)
    ├─⑥ 法人サブスクレポート自動生成(L2: 送付前承認)
    ├─⑦ OEM診断ウィジェット(iframe/API提供)
    ├─⑧ SNS投稿・画像生成(デザイン自動化パイプライン転用、L2承認)
    └─⑨ 週次KPI・月次レビュー資料・受給統計の自動生成(アカリ→達成くん形式)
[統制] hooksで守り部ゲート強制 / タダス(独立監査)が全ログを定期監査 / 事故→テスト化の手順
[人間] 金曜承認バッチ + 面談 + 決裁上申(それ以外はやらない)
```

## 第5部 M1に前倒しする「30日構築順序」(最初の1ヶ月のClaude Code作業)

- **Week 1**: フクギイロ用リポジトリ整備。CLAUDE.md憲法(本3文書のルールをエージェント憲法化)・部門エージェント定義の増設。hooks守り部ゲート実装
- **Week 2**: 収集パイプライン移植(国の個人向け制度→構造化→DB)。ヘッドレスcron稼働
- **Week 3**: 検証システム構築(生成/検証分離・ゴールデンセット・CI回帰テスト)。**ここまで終わるまで公開しない**
- **Week 4**: サイト初版+制度50件公開(L2承認フロー込み)。週次KPIレポ自動化。M2の5市クローラー着手

## 第6部 この設計が担保する「確実性」の根拠(まとめ)

1. 技術基盤(cron→Claude→静的公開)は**hojo-hqで既に本番稼働しており**、移植であって新規開発ではない
2. 収益構造(利益率90%型・限界費用ゼロ)は世界で複数の実例が公開収益で証明済み
3. 集客戦術(無料ツール・programmatic SEO・build in public)は再現手順が確立した定石のみ採用
4. 事故防止(eval・自律度ダイヤル・独立監査)は2026年のエージェント運用標準に準拠
5. 唯一世界に前例がないのは「受給完了まで追う」ことだが、これは競合不在の空白であり、技術的には受給報告フォーム+カウンターという単純な実装で成立する
