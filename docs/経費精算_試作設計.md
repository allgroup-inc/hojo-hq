## 0. 設計五原則(全判断の物差し)

| 原則 | 具体的な基準 |
|---|---|
| 使いやすさ | 社員の動作は**2つだけ**: ①撮って送る ②10秒補記。3タップ目が必要な設計は差し戻し |
| 美しさ | ブランド準拠(ネイビー#00335C・オレンジ#F88800・Noto Sans JP)。全接点(受付返信・一覧・通知)で文言と見た目を統一 |
| わかりやすさ | 専門用語ゼロ。「仕訳」「電帳法」等はシステムの裏に隠し、画面には出さない。中学生が読める文言のみ |
| 対応工数の少なさ | **カスタムアプリを作らない**。M365標準機能+ビュー整形+フロー1本で構成。作る前に「その機能は要るのか」を問う(カイゼンさん基準) |
| 管理のしやすさ | 管理者が見る場所は**1画面**(「要対応」ビュー)。設定・文言・フロー定義はすべて本リポジトリで管理し、引き継ぎ台帳に登録 |

## 1. 社員から見た体験(ここがすべて)

```
①【撮って送る】 領収書をスマホで撮影 → keihi@ にそのまま送信(件名・本文不要)
      ↓ 1〜2分後
②【お礼が返る】 「お預かりしました」返信が届く
      ・読み取り結果(日付・お店・金額)がカードで表示
      ・「あとは 目的 を選ぶだけです」と次の一歩を案内
      ↓
③【10秒補記】 返信内のリンクを1タップ → 自分の一覧が開く
      ・目的: 選択チップ(移動/打合せ/会食/備品/その他)
      ・会食のときだけ「どなたと」欄が現れる(不要な欄は最初から見せない)
      ・金額を目視確認して「これで申請」ボタン
      ↓
④【進捗が見える】 承認されたら「承認されました。今月の精算は◯日に振り込まれます」
```

### ホスピタリティ実装(パネル監修の反映点)
- **二度聞かない**(オカミさん): 送信者アドレスで本人特定。氏名・所属・日付の入力は一切求めない。よく行く店は費目を学習し次回から自動提案
- **Noと言わない**(コンシェルジュさん): 読み取り失敗時も「読み取れませんでした。やり直してください」とは言わない。「お預かりしました。こちらで確認して入力します」→ 失敗分は管理側の「要対応」に回り、**社員側の体験は絶対に止めない**。ピンボケでも撮り直し指示は最大1回まで
- **文言の所作統一**(キャビンさん): 全通知は「お預かり→お礼→進捗→完了報告」の型。催促文はコンシェルジュ調(「月末が近づいてまいりました。溜まっていた1枚、いまなら10秒で終わります」)。エラー・却下の文言は責めない言い回しのみ

## 2. 裏側の構成(社員には見せない水面下)

```
共有メールボックス keihi@
  → Power Automate(フロー1本のみ)
      1) 添付を SharePoint ライブラリへ保存(原本・自動命名: 日付_送信者_連番)
      2) Claude API で読取(日付/支払先/金額/税率/インボイス登録番号/確信度)
      3) SharePoint リスト「経費精算」に1行追加(送信者=申請者に自動紐付け)
      4) 本人へ「お預かり」返信(結果カード+マイページリンク)
      5) 低確信度・読取失敗 → ステータス「要対応」(本人には失敗を見せない)
  → 承認: 上長にメールで届く「承認/差戻し」1タップ(Power Automate標準承認)
  → 月次: 仕訳CSVを自動生成し経理へ(画面には出さない・裏の仕事)
```

- 一覧の見た目は **SharePointビュー整形(JSON)** で作る: ステータスをブランド色のバッジ表示、金額を右揃え等幅、行間広め。カスタムアプリ(Power Apps)は**作らない**(工数と保守の観点で禁止。ビュー整形JSONも本リポジトリで管理)
- マイページ=「申請者=自分」で絞ったビュー+アイテムレベル権限(作成者のみ編集)。経理は全件ビュー

## 3. 管理者(経理・たかしくん)の体験

- 見る場所は「**要対応**」ビュー1画面のみ: 読取失敗・差戻し・期限超過が全部ここに集まる。ゼロなら何もしない日
- 週次で自動サマリー1通: 「今週◯件・要対応◯件・平均処理時間◯分」(目で見る管理・アンドン方式)
- 文言・費目・承認者の変更はリストの設定1か所。フロー改修不要の範囲を最大化

## 4. 三名体制議事(社外ホスピタリティ・パネル)

【経費精算v2 体験設計】(見直し期限: 2027-01-23)
- **スイシン(担当: オカミさん)**: 「二度聞かない・水面下で済ませる」を徹底すれば、社員は精算を“作業”と認識しなくなる。定着率が上がれば経理の締め作業も平準化し、五原則すべてが同時に立つ
- **ウタガイ(担当: コンシェルジュさん)** ※必須: ①「失敗を本人に見せない」設計は、裏の要対応が滞留すると**本人が申請したつもりで未処理**という最悪の体験を生む。②美しさを追いPower Appsに手を出すと保守が重くなり五原則の「工数最小」と矛盾する。③お礼メールが毎回届くのは3か月で「通知疲れ」に変わる恐れ
- **ベッカイ(担当: キャビンさん)**: そもそもメール添付が最短か。LINE WORKSに慣れた組織なら「トークに写真を送るだけ」が動線として1タップ短い。またそもそも論として、法人カード比率を上げれば撮影自体が減る(機内サービスと同じで、提供回数を減らすのが最上の効率化)
- **裁定(起案者たかしくん整理・最終決裁は小柳さん)**: スイシン案を骨格として採用。ウタガイ①は「要対応48時間ルール」(48時間超は管理者へ自動エスカレーション+本人へ正直に状況連絡)で処理。ウタガイ②は五原則に「カスタムアプリ禁止」として明文化済み。ウタガイ③は通知を「お預かり」と「承認完了」の2通に絞り、中間経過は送らない。ベッカイのLINE WORKS入口はPhase 2の第一候補として残し、法人カード化は経理と別線で検討(本設計のスコープ外だが小柳さんへ論点上申)

## 5. 法務・税務要件(守り部レビュー必須・v1から変更なし)

紙原本を破棄しスキャン画像を正本にするには電子帳簿保存法のスキャナ保存要件を満たす必要がある。
1. 改ざん防止: 「訂正削除の防止に関する事務処理規程」整備(国税庁ひな形)+SharePointバージョン履歴
2. 検索要件: 日付・金額・取引先で検索可(リスト列で充足)
3. 入力期限: 受領からおおむね7営業日以内に送信するルール(催促はコンシェルジュ調リマインドで)
4. 画質: 200dpi相当・カラー(スマホで通常充足)
- **顧問税理士の確認完了までは紙原本を併存保管**。インボイス登録番号なしは免税事業者フラグで経理へ
