# フクギイロ 計測・レポート設計 — 「実感」を数字で見る基盤

作成日: 2026-07-23 / 決裁根拠: 小柳さん方針(2026-07-23)「マネタイズは県民が実感してから。県民からは一切取らない。まず県民10%=14万人の登録」
目的: いつ・何人が・どこを見て・どこで離脱し・登録/申請し・**実際に受け取れたか**を、属性(年齢帯・地域・性別)別に常時レポートできる基盤の設計。

---

## 0. 大原則(計測にも「守り」を)

1. **個人を追わない。集団を見る。** 個人の行動履歴を売る/使うためではなく、「どこで県民がつまずくか」を直すための計測
2. **匿名集計のみ公開・共有**: 集計セルが10人未満になる切り口は出さない(小さな村×年齢帯で個人が特定される事故の防止)
3. 取得項目の追加は必ずマモリさん審査+小柳さん決裁(現在の診断は市町村・世帯構成・年齢帯・収入帯(任意)のみ。**性別は現在取得していない** — 追加する場合は「任意・答えなくても診断できます」方式で、統計目的を明示して同意を得る)
4. 計測ツールもCookieレス・国内外への無断送信なしを守り部審査の条件とする

## 1. 何を測るか(ファネル全段の定義)

```
[知る]   SNS表示 → クリック(UTM別: instagram/facebook/line/検索/紹介)
[見る]   LP・市町村ページ・制度ページの閲覧(時間帯・地域ページ別)
[試す]   診断開始 → 各質問の通過/離脱(Q1〜Q5のどこで抜けたか) → 診断完了
[登録]   LINE登録(診断完了→登録率30%目標)
[動く]   書類チェックリスト閲覧 → 「申請しました」報告
[受かる] 「振り込まれました」報告(制度名・金額・任意で かかった費用)
[実感]   知ってから入金までの日数(制度別・市町村別の中央値)
```

## 2. 属性別レポートの切り口

| 軸 | 粒度 | 取得元 |
|---|---|---|
| 地域 | 市町村(41)| 診断Q1 / 市町村ページ閲覧 |
| 世帯 | 子の有無・年齢帯 | 診断Q2-3 |
| ライフイベント | 11分類 | 診断Q4 |
| 収入帯 | 5区分(任意回答のみ) | 診断Q5 |
| 年齢・性別 | LINEアンケート(任意)で将来取得 | LINE登録後(要マモリ審査) |
| 時間 | 時間帯・曜日・月 | イベントタイムスタンプ |

## 3. 「実際に受け取れたか」の蓄積スキーマ(受給ジャーニー)

LINE登録者単位で以下を記録(LINE開設後に実装。個人名不要、LINE識別子のみ):

| 項目 | 例 | 取得方法 |
|---|---|---|
| 診断日・該当制度 | 2026-08-01・児童扶養手当 | 診断結果の保存(本人同意) |
| 申請日 | 2026-08-10 | 「申請しましたか?」への返答 |
| 入金日 | 2026-09-25 | 「振り込まれましたか?」への返答 |
| 受給額 | månad◯円/一時金◯円 | 受給報告(自己申告) |
| かかった費用(任意) | 書類取得550円・交通費・専門家費用 | 受給報告の任意項目 |
| つまずき(任意) | 「書類が分からず窓口2回」 | 自由記述→分類タグ化 |

→ ここから出せる指標: **制度別の「知ってから入金までの日数」「実質手取り(受給額−費用)」「挫折率と挫折理由」**。この3つは日本のどこにも存在しないデータであり、行政への政策提言と診断改善の両方の燃料になる。

## 4. 段階実装(できることから正直に)

| 段階 | 時期 | 実装 | 取れるもの |
|---|---|---|---|
| **第1段階(現在・稼働中)** | 今 | 端末内計測(localStorage)+UTM+診断の質問別到達 | 診断開始/完了・離脱位置(本人の端末内のみ。集計不可) |
| **第2段階(要決裁)** | 決裁後即 | Cookieレス計測ツール導入(守り部審査済みのもの) | 時間帯別訪問数・ページ別閲覧・離脱率・流入元別CV — **「何時にどのくらいの人がどこを見たか」はここで実現** |
| **第3段階(LINE開設後)** | LINE後 | 受給ジャーニー(§3)+属性アンケート(任意) | 申請率・入金までの日数・実質手取り・属性別分析 |
| **第4段階(定常)** | M6〜 | 週次レポート自動生成(アカリ形式に統合)+月次「沖縄もらい忘れレポート」公開 | 全指標の自動ダッシュボード化 |

- 第2段階のツール選定はマモリさん審査(Cookieレス・データ国内/EU保管・無断第三者提供なし)→小柳さん決裁。候補比較表を次アクションで作成
- レポートは週次自動(ユイさんの週次レポに統合)。すべての数字は「先週比」と「ボトルネック1つ」つきで出す(数字の羅列にしない)

## 5. 北極星の更新 — 「実感ファースト」の指標体系

小柳さん方針を受けて、指標の序列を明文化する:

```
第0階層(すべての上位): 受給実感 = 実受給額累計・受給世帯数・「助かった」の声
第1階層: 登録 = LINE登録者数(最終目標: 県民の10% ≒ 14万人)
第2階層: 行動 = 診断完了数・申請報告数
第3階層: 認知 = 訪問・SNSリーチ
収益はこの序列の「結果」であり、目標ではない(収益目標が実感目標より先に立ったら設計ミス)
```

- 14万人への現実的な階段: 1年目1万人(登録者。うち支援世帯1,000)→3年目5万人→5年目14万人(県民10%)。年次で見直し
- **審査付き広告モデル(小柳さん発案)を出口候補に正式登録**: 14万人が見るサイトの広告枠を「県民がプラスになる商品を持つ企業のみ」に販売する。実装条件: ①掲載可否の審査基準を守り部+三名体制で策定 ②CV単一原則(絶対ルール4)の改定として小柳さん決裁 ③実感指標(第0階層)が十分に積み上がった後。**広告が先、実感が後、の逆転は絶対にしない**

## 6. 三名体制議論の記録

- スイシン: 第2段階(計測ツール)を最速で決裁に上げる。時間帯・離脱・地域別は現状ゼロなので導入効果が最大
- ウタガイ: 属性(性別・年齢)の取得欲は「分析の快楽」に流れやすい。**「その属性軸がないと直せない改善は何か」を先に書けない項目は取得しない**ルールを提案 → 採用(§0-3に反映)
- ベッカイ: 受給ジャーニーはLINEを待たず、診断結果画面に「あとで結果をメールで受け取る」等の代替もある → 不採用(メールアドレスは取得最小化に反する。LINE一本化を維持)。ただし診断結果画面に「この結果を保存(端末内)」は追加検討
- 裁定(ユイ): 本設計で確定。次アクションは①計測ツール比較表(マモリ審査用)②受給報告スキーマのLINE実装仕様化
