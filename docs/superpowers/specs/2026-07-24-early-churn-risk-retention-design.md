## 1. 目的(何を作るか)

申込〜解約の実績データを蓄積・反映し続け、**新しい申込みが入った時点で「6ヶ月以内に解約する可能性(%)」**を算出する。
高リスクにヒットした先へ**先回りで手厚い保全**をあて、早期解約を減らし継続率を高める。

一言でいうと:
> 申込〜解約のデータを常に反映し、新規申込みに「早期解約リスク◯%」と「なぜ高いか(ヒット要因)」を出す。高リスク先に先回り保全をあて、効果を数字で検証する仕組み。

### 確定した前提(ブレストで合意)
| 項目 | 決定 |
|---|---|
| データの現状 | 統一エクセル・数千件以上(解約済み含む) |
| 「早期解約」の定義 | **申込(契約)から6ヶ月以内の解約** |
| 出力 | 一覧 → カード → レポートの3種。土台(一覧)から段階出荷 |
| 計算方針 | **A案:実績ベースの要因別リスク率スコアリング**(説明可能・自己更新)。将来B案(機械学習)へ差し替え可能な構造にする |
| 顧客データの扱い | **コードだけ共有・顧客個人データはローカル厳守(リポジトリに絶対コミットしない)** |

## 3. アーキテクチャ(4段パイプライン)

```
① 取込    統一エクセル(ローカル) → 内部の正規化テーブルに整える
② 学習    成熟した実績から「要因別6ヶ月解約率」を計算 → risk_model.json に保存  ← "常に反映"
③ 採点    継続中×6ヶ月未満の申込みに、該当要因を合成して 総合リスク% + ヒット要因 + リスク帯
④ 出力    一覧 / カード / レポート  +  リスク帯ごとの推奨保全アクション
```

②を回し直すだけでモデルが最新化される(「エクセル更新 → 再実行」で常にデータ反映)。各段は独立した部品として単体テスト可能にする。

### 技術方針
- 純Python + JSON(既存リポジトリの作法に準拠。追加インフラ不要)。エクセル読取は標準的なライブラリ(openpyxl/pandas 等、実装計画で確定)。
- 各段は独立モジュール:`intake` / `fit` / `score` / `report`。
- 将来B案(scikit-learn 等)へは `fit`/`score` の内部だけ差し替え、取込・出力層は不変にする。

## 5. データスキーマ(内部の正規化テーブル)

エクセルの実列 → 内部項目の対応づけ設定(`column_map`)を1枚かませ、列名の揺れを吸収する(パイプさん担当)。

### 入力属性(リスクの"要因")
| 内部項目 | 中身 | 備考 |
|---|---|---|
| `apply_id` | 申込ID | 顧客名は使わずIDのみ |
| `apply_date` | 申込(契約)日 | 経過月数・成熟度の起点 |
| `product` | 商品・プラン種別 | 保険/家計/催事など |
| `channel` | 集客チャネル(流入元) | 催事・紹介・SNS・Web等 |
| `apply_form` | 申込形態 | 対面/オンライン/書面 等 |
| `amount` | 金額(月額 or 一時金) | 単位を正規化 |
| `age_band` | 年代 | 生年月日/年齢を20代・30代…にビン化 |
| `gender` | 性別 | |
| `area` | 地域 | 市町村→エリアに正規化 |
| `agent_id` | 営業マン | ID運用。評価誤用防止は監査コウヘイさん |

### 結果(アウトカム)
| 内部項目 | 中身 | ロジック |
|---|---|---|
| `cancel_date` | 解約日 | 空なら継続中 |
| `cancel_reason` | 解約理由 | レポートの分析軸 |
| `months_elapsed` | 経過月数 | 解約日 or 今日 − 申込日 |
| `is_mature` | 成熟フラグ | 申込から6ヶ月以上経過したか |
| `is_early_churn` | 早期解約フラグ | 6ヶ月以内に解約=1、それ以外=0 |

### 数字を嘘にしない扱い(打ち切り/成熟度)
- **学習(②)に使う集合**:`is_mature = Yes`(6ヶ月経過済み)のみ。未成熟の継続中を「解約しなかった」に数えない。
- **採点(③)の対象集合**:継続中 × 6ヶ月未満(=これから手を打てる人)。
- 役割の違う2集合を明確に分ける。

## 6. リスク%の計算ロジック(A案の中身)

「全体平均解約率」を出発点に、要因ごとの倍率(リフト)を実績から求めて合成し、過去データで較正して実際の確率に合わせる。

### 手順
1. **ベース解約率**=成熟実績全体の6ヶ月以内解約率(例:12%)。
2. **要因別の倍率(リフト)**を実績から算出。
   | 要因(値) | 実績解約率 | 倍率 |
   |---|---|---|
   | 商品=X | 20% | ×1.7 |
   | 集客=催事 | 18% | ×1.5 |
   | 年代=20代 | 15% | ×1.25 |
   | 申込形態=対面 | 10% | ×0.85(下げる) |
3. 新規申込みの該当倍率を合成 → **総合リスク%**。過去データで較正(出しすぎ/出さなすぎを補正)。例:約30%(高リスク)。
4. **寄与の大きい要因トップ3を「ヒット要因」**として表示(なぜ高いかを言語化)。

### 数字を嘘にしない3つの仕掛け
- **少件数のスムージング(ベイズさん)**:実績が少ない要因は倍率を全体平均へ引き寄せる。「3件で高リスク断定」を禁止。母数の少ない要因は「参考値」と明示。
- **バックテストで較正(ケンテイさん)**:過去時点で採点していたら当たっていたかを検証。当たらないまま本番に出さない(QCMさんの品質ゲート)。
- **掛け合わせの手当て(ミトオシ部門)**:組合せで跳ねる要因(例:催事×20代×低単価)は実績で確認し、重要な組合せだけ個別ルール化。

### 監査の関与(独立)
- ダトウさん:母数不足での断定・偏った判定の統計監査。
- コウヘイさん:営業マン別倍率の人事評価誤用・自己成就予測の検知。

## 7. リスク帯・保全アクション・出力

### リスク帯(初期目安)
| 帯 | 目安 | 意味 |
|---|---|---|
| 🔴 高 | 上位 約20%(または実績解約率が概ねベースの2倍以上) | 先回り保全を必ずあてる |
| 🟡 中 | 高と低の間 | 標準フォロー+「様子見放置」を監視 |
| 🟢 低 | 下位 | 通常運用 |

閾値は省略不可の3役議論案件。「手厚い保全に回せる人数」に合わせて調整(トリアージの重症度分類=全員は救えない を借用)。

### 手厚い保全アクションの型(テアツ部門)
| 帯 | 標準アクション | 担当 | タイミング |
|---|---|---|---|
| 🔴 高 | 申込直後の安心コール+3ヶ月までの定期接触+ヒット要因に応じた個別フォロー | 保全担当/営業マン | 申込〜7日以内に初回 |
| 🟡 中 | オンボーディング配信+1回フォロー。改善なければ高扱いへ | CS(ツヅケさん設計) | 申込〜1ヶ月 |
| 🟢 低 | 通常運用 | ― | ― |

> ※この保全連絡は制度の締切アラート(SNS/LINE個別/次回予告の3層ルール)とは**別系統**。混同しない。

### 出力3種(段階出荷)
1. **一覧(土台・最初に作る)**:継続中×6ヶ月未満をリスク高い順に並べた表。帯・%・ヒット要因・推奨アクション・担当。営業マン別/商品別で絞り込み。
2. **カード**:申込1件の詳細(%・ヒット要因トップ3・なぜ高いか・型アクション)。
3. **レポート**:営業マン別/チャネル別/商品別の解約傾向と改善推移。QCMさんが週次で達成くんフォーマットに反映。

### 効果測定(やりっぱなしにしない)
高リスク帯で保全をあてた群と、あてられなかった群の**その後の実際の6ヶ月解約率を比較**。改善が数字で見えなければアクションの型を見直す(ケンテイ+監査)。

## 8. スコープと段階(フェーズ)

| フェーズ | 内容 | 完了の目安 |
|---|---|---|
| P0 | 実エクセルの列確認 → `column_map` 作成。`private/` + gitignore 整備 | 取込が再現的に通る |
| P1(土台) | ①取込 ②学習 ③採点 ④一覧出力。ベース率+要因別リフト+スムージング+バックテスト | 一覧が出て、バックテストで妥当性確認 |
| P2 | カード出力 | 1件詳細が出る |
| P3 | レポート出力+効果測定の枠 | 週次レポに載る |
| 将来 | 掛け合わせ個別ルール強化 / B案(機械学習)への差し替え検討 | 精度が頭打ちになったら判断 |

### やらないこと(YAGNI)
- 初手から機械学習(B案)には行かない。まず説明可能なA案で土台と検証を作る。
- 顧客個人データのクラウド化・外部連携は本スコープ外(将来、守り部審査を経て別途)。
- 締切アラート(制度告知)系との統合はしない(別系統)。
