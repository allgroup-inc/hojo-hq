# 技術ロードマップ — アーキテクチャ監査と段階移行計画

作成: 2026-07-23 / 起点: [引き継ぎ台帳](引き継ぎ台帳.md) / 決裁: 小柳さん
監査の問い: **1,000社登録・企業台帳蓄積・個別通知・完全自動化・100点基準に対して、現構成(GitHub Pages+Actions+PIL+手動SNS)は最適か?**

---

## 結論(先に一言)

**公開情報サイトとしての現構成はほぼ最適。ただし事業の本命(❺診断×LINE ID紐付け・企業台帳・個別通知)は、静的サイト構成では原理的に実現不可能。**
よって取るべき道は「置き換え」ではなく「**非公開バックエンドの増築**」。GitHub Pages+Actionsはそのまま公開面として残し、裏側にLIFF+軽量バックエンド+台帳ストレージを足す。初期費用ほぼゼロで❺❼❹が埋まり、月額約7,000円まで許容すれば1,000社スケールまで見通せる。

---

## ① 現構成の率直な評価

### 強み(手放すべきでないもの)

| 資産 | 評価 |
|---|---|
| GitHub Pages 公開 | 月額0円・CDN配信・実質無限のトラフィック耐性。沖縄の中小向け情報サイトとして性能は過剰なほど十分 |
| Actions 1日4回パイプライン | fetch→整形→画像生成→デプロイが完全自動。個人運用でここまで組めている例は少ない |
| 日次ヘルスチェック+Issue自動起票/自動クローズ | ❶❼の土台がすでにある。監視の「検知」側は完成度が高い |
| PIL画像生成 | ブランド決定版デザインをCI内で量産。外部サービス依存ゼロ・費用ゼロ |
| リポジトリ=単一の真実 | データ・コード・文書・監査記録が1箇所。引き継ぎ耐性が高い |

### 限界(静的構成の宿命 — 努力では超えられない壁)

1. **ユーザー識別ができない**: 静的HTMLは「誰が診断したか」を知る術がない。❺(診断×LINEユーザーID紐付け)は現構成の延長線上に存在しない
2. **非公開ストレージがない**: 公開リポジトリに企業情報は置けない(法務ルール絶対)。台帳の置き場所が現構成にはない
3. **プッシュ通知が打てない**: 「貴社が使えます」個別通知(本命機能)も、締切7〜29日のLINE個別アラートも、送信基盤が存在しない
4. **計測ゼロ**: 診断→登録率30%というKPIが今は測定不能。改善サイクルが回らない
5. **Actions cronの遅延**: GitHub cronは混雑時15分以上遅れる/稀にスキップされる。日次バッチには十分だが、分単位の正確性が要る用途には不向き
6. **手動SNS**: 投稿文・画像は自動生成済みなのに、最後の「投稿」だけ人間。ここだけ自動化が切れている
7. **E2Eの「通し」がない**: 現ヘルスチェックはサイト側の点検。❶が求める「診断→登録→相談→返信」の通し検証は、そもそも登録・返信の裏側が存在しないため検証対象自体が未完成

## ② 推奨アーキテクチャ(段階移行)

### 全体像

```
[公開面 — 現構成を維持]
GitHub Pages(サイト+診断UI) ← Actions 1日4回(jGrants+県収集→Claude API整形→PIL画像)

[増築する非公開面 — Phase 1で新設]
診断ページをLIFF化 → LINEユーザーID+診断回答を取得
        ↓ POST
GAS Webhook(無料) → 非公開Googleスプレッドシート=企業台帳
        ↓ 毎日
マッチングジョブ(GAS or Actions) → LINE Messaging API push
   ├ 締切7〜29日アラート(個別)
   ├ 「貴社が使えます」通知(診断データ×新着)
   └ 障害時: 小柳さんへ即時通知
```

### 移行フェーズと費用

| フェーズ | 内容 | 月額 |
|---|---|---|
| **Phase 0(即日)** | GA4+Clarity導入 / 障害LINE通知 / Lighthouse CI | ¥0 |
| **Phase 1(〜2週間)** | 診断LIFF化+GAS+Sheets台帳(❺) / Claude API原文照合(❹) | ¥0〜数百円 |
| **Phase 2(〜2ヶ月)** | 個別通知エンジン / Meta Graph API予約投稿 / Playwright通しE2E(❶) | 約¥5,500〜7,000 |
| **Phase 3(登録600社〜)** | 台帳をSupabaseへ移行 / 配信量に応じLINEプラン増強 | 約¥11,000〜 |

### 【必須評価】各連携候補の評定

**LINE Messaging API / LIFF — ★採用(事業の心臓部)**
- LIFF(無料): 診断ページをLIFFアプリとして開くだけで `liff.getProfile()` からLINEユーザーIDが取れる。❺の「自動紐付け」はこれ一択。サイト改修は診断ページへのSDK追加のみで、現UIはほぼ流用可
- Messaging API(従量): push送信が個別通知の唯一の正攻法。**LINE Notifyは2025年3月終了済み**のため、障害通知も本APIに一本化する
- 料金: コミュニケーションプラン無料200通/月 → 登録数百社で個別アラートを打つ頃にライトプラン(5,000通/月・約¥5,500)へ。**2026年10月に料金改定が予告されている**ため、Phase 2着手時に最新料金を要再確認
- 応答モード注意: 現在チャットON運用。Webhook自動応答と併用する場合はチャットと自動応答の切替設計が必要(当面はpush専用で衝突なし)

**非公開データ基盤 — ★Phase 1はGAS+Sheets、Phase 3でSupabase**

| | GAS+Sheets | Supabase | Cloudflare Workers+D1 |
|---|---|---|---|
| 月額 | ¥0 | ¥0(無料枠)→$25 | ¥0(無料枠)→$5 |
| 実装難度 | 低(JSのみ・サーバ管理不要) | 中 | 中〜高 |
| 非エンジニアの閲覧 | ◎ スプレッドシートをそのまま共有 | △ 管理画面 | ✕ ほぼ不可 |
| スケール上限 | 実行6分/回・同時実行弱い。数千行なら余裕 | Postgres・実質無制限 | SQLite・十分大きい |
| 落とし穴 | 大量同時アクセスに弱い | 無料枠は1週間非アクセスで休止 | 台帳UIを自作する羽目に |

- 判断: 台帳は「たかしくん・提携部が日々見て営業に使う」データ。**スプレッドシートがそのまま営業ツールになる**GAS+Sheetsの業務価値が初期は圧倒的。1,000社×診断回答でも数千行であり性能問題は起きない
- 移行トリガー(どれか1つで Supabase へ): 登録600社超 / マッチング条件が複雑化しSQLが欲しくなる / GASの6分制限に当たる。Sheets→Postgresは機械的に移行可能なので、初手GASは技術的負債にならない

**Claude API — ★採用(すでに整形で使用中。3用途に拡張)**
1. **原文照合の自動サンプリング(❹)**: 毎日5件、掲載データと出典URL原文を突合し差異をIssue起票。Haiku 4.5($1/$5 per MTok)なら月数十円
2. **「貴社が使えます」マッチ理由文の生成**: 診断回答×制度要件から個別通知の理由文を生成。テンプレ通知との差が登録維持率に直結
3. **自治体PDF公募要領の構造化**: fetch_localの拡張。PDFしか出さない市町村の制度を取り込める(掲載150件以上の目標に効く)
- 月額見込み: 全部合わせて数百円。費用対効果は全候補中で最高

**Meta Graph API(Instagram/FB予約投稿) — ★採用だが2段階で**
- 技術的には無料で完全自動化可能: IGビジネスアカウント+FBページ連携+Metaアプリ(`instagram_content_publish`権限)でActionsから投稿API実行
- ただしアプリ審査・トークン管理の初期ハードルがある。**先にMeta Business Suiteの予約投稿(手動・無料)で運用を回し、投稿フォーマットが安定したPhase 2でAPI化**が現実的。台帳の未完了タスク「Meta Business Suite連携」がそのまま前段になる
- 注意: 画像は公開URLが必要 → 現構成でposts/imagesがPagesに乗っているのでそのまま使える

**計測(GA4/Clarity)+障害通知 — ★即日採用(両方無料)**
- GA4: 診断開始→完了→LINEタップのファネル計測。UTM規律(SNS部)とセットで「どのチャネルが登録を生むか」が初めて見える
- Microsoft Clarity: ヒートマップ+セッション録画が無料。❸の「初見5人テストで詰まりゼロ」の証拠収集にそのまま使える
- 障害通知(❼): healthcheck.ymlのfailureステップにMessaging API pushを1ステップ追加 → 小柳さんのLINEへ即時通知。無料枠200通で余裕

**Canva API — ✕見送り(率直な評価)**
- Connect APIはPro(月$15≒¥2,000強)前提だが、**現PILパイプラインがすでに決定版デザインを全自動で量産できている**。APIで得られるのは「Canvaテンプレの柔軟性」だが、それはデザイン変更頻度が高い場合の価値。ブランド確定済みの現状では費用対効果が合わない
- 人間がデザインを触る作業(バナー・催事チラシ等)にはCanva Pro単体の価値はあるが、それはAPI連携とは別の話。再検討条件: デザイン刷新が月2回以上発生するようになったら

**その他(知られていない可能性のある選択肢)**
- **Lighthouse CI(無料)**: ❷「4項目95以上」をActionsで毎日自動計測・閾値割れでfail。手動計測を今すぐやめられる
- **Playwright(無料)**: ❶の通しE2E。実ブラウザで診断→LIFF→(テスト用ID)登録→push受信までを毎日検証
- **cron-job.org等の外部cron(無料)**: GitHub cronの遅延が問題化した場合の起動代替(webhook dispatch)
- **Lステップ/エルメ等のLINEマーケSaaS**: セグメント配信UIは強力だが月額1万〜数万円。自前のGAS+Messaging APIで同等機能を無料実装できるため、運用を非エンジニアへ完全移管する日まで不要
- **BigQuery連携(GA4標準・無料枠)**: 将来、台帳×行動ログの突合分析が欲しくなったときの受け皿

## ④ 最初に着手すべき3件

**1. 障害LINE通知(❼)— 所要半日・¥0**
healthcheck.ymlのfailure時にMessaging API pushで小柳さんへ即通知。既存のIssue起票機構に1ステップ足すだけで100点基準が1つ埋まる、全候補中で最短の勝ち筋。ここでMessaging APIのチャネル発行・トークン管理を経験しておくことが、そのまま❺・個別通知の下準備になる。

**2. 診断LIFF化+GAS+Sheets企業台帳(❺)— 所要1〜2週間・¥0**
最重要。1,000社登録は「数」ではなく「台帳という資産」になって初めて意味を持ち、本命機能(個別通知)も出口(GLOW接続)もすべてこの上に建つ。着工が1ヶ月遅れれば、その間の登録者は全員「IDと診断が紐付かない過去分」になり後から回収できない。法務ルール(非公開ストレージ必須)を満たす唯一の道でもある。

**3. GA4+Clarity導入(計測)— 所要半日・¥0**
「診断→登録率30%」がKPIなのに現在は測定不能。初回投稿(未完了タスク)でSNS流入が始まる前に計測を入れておかないと、最初の貴重な流入データが永遠に失われる。改善はすべて計測の後にしか来ない。

## 付記: コストの全体観

12ヶ月目(1,000社・個別通知フル稼働)時点の想定月額は **約¥11,000**(LINEライト¥5,500+Supabase $25+Claude API数百円)。リード獲得単価に換算すると月10件の面談に対し¥1,100/件であり、GLOW案件1件の価値に対して事実上誤差。**技術コストがこの事業のボトルネックになることはない。ボトルネックは常に「❺の着工時期」である。**
